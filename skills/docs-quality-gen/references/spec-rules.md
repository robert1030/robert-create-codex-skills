# SPEC Rules

## Purpose

Use these rules when producing, updating, or reviewing SPEC-style documents for robert's personal documentation workflow.

A SPEC is a contract. It says what must be true, what must be supported, and what can be verified. It should not become a step-by-step runbook.

## Core Rules

- Write requirements as things that can be checked when possible.
- Keep operation steps in the runbook, not in the SPEC.
- Mark version numbers as either current version facts or lasting requirements.
- Do not write local paths, user names, drive letters, or tool install locations as universal requirements.
- If a local environment example is useful, label it as an example or current environment and tell readers to replace it on other computers.
- Mark observed behavior as observed behavior. Do not present it as an official rule unless the source proves it.
- Make examples broad enough that readers do not mistake them for the only supported condition.
- Keep terminology consistent across SPEC, runbook, and README.

## Readability Rules

- Bilingual headings are acceptable when they help scanning.
- Add a short Chinese explanation before a large technical section when the main point could be missed.
- Explain a technical term the first time it appears.
- Keep precise technical terms when they matter, but make the main idea understandable first.
- Separate formal requirements from examples and observations.
